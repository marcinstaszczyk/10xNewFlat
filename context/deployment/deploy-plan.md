# First Cloudflare Workers Deployment Plan

## Summary

- Deploy the Astro 6 SSR app to Cloudflare Workers, not Cloudflare Pages.
- First production URL: the default `workers.dev` URL.
- Supabase production credentials are required before deployment.
- Local preflight already passed: `pnpm.cmd run lint`, `pnpm.cmd run build`, and `wrangler deploy --dry-run`.
- First Worker creation must use `wrangler deploy`; `wrangler versions upload` cannot create the initial Worker.

## Step 0: User Actions

- Ensure Cloudflare access is ready by either logging in with Wrangler or setting a scoped `CLOUDFLARE_API_TOKEN`.
- Create or choose the production Supabase project.
- Prepare the production values for `SUPABASE_URL` and `SUPABASE_KEY`.
- Approve the agent to update `wrangler.jsonc` and run deployment commands.
- After deploy returns the `workers.dev` URL, update Supabase Auth URL Configuration:
  - Site URL: exact deployed `https://<worker>.<subdomain>.workers.dev`
  - Local redirect URLs stay only for local development.

## Implementation Steps

- Update `wrangler.jsonc`:
  - Change `name` from `10x-astro-starter` to `10x-new-flat`.
  - Keep Workers config, `output: "server"`, and Cloudflare adapter unchanged.
- Use these runtime secrets:
  - `SUPABASE_URL`
  - `SUPABASE_KEY`
- Use the Cloudflare bindings generated by the Astro build:
  - `ASSETS`
  - `SESSION` KV
  - `IMAGES`

## Deployment Commands

Use Windows-safe local Wrangler execution:

```powershell
$env:HOME = ".wrangler-home"
$env:USERPROFILE = ".wrangler-home"
$env:APPDATA = ".wrangler-config"
$env:LOCALAPPDATA = ".wrangler-cache"
$env:XDG_CONFIG_HOME = ".wrangler-config"
$env:XDG_CACHE_HOME = ".wrangler-cache"
$env:WRANGLER_LOG_PATH = ".wrangler-logs"
$env:WRANGLER_SEND_METRICS = "false"
$wrangler = ".\node_modules\.bin\wrangler.CMD"
```

Verify Cloudflare authentication:

```powershell
& $wrangler whoami --json
```

Store production secrets in ignored `.env.production`:

```dotenv
SUPABASE_URL=https://<project-ref>.supabase.co
SUPABASE_KEY=<anon-key>
```

Run gates:

```powershell
pnpm.cmd run lint
pnpm.cmd run build
& $wrangler deploy --dry-run --secrets-file .env.production
```

Deploy after human approval:

```powershell
& $wrangler deploy --secrets-file .env.production --message "initial production deploy"
```

## Verification

- Open `/`, `/auth/signin`, and `/auth/signup` on the deployed URL.
- Confirm `/dashboard` redirects unauthenticated users to `/auth/signin`.
- Create a test user, confirm email if required, sign in, and verify `/dashboard`.
- Sign out through `/api/auth/signout`.
- Check runtime logs:

```powershell
& $wrangler tail --name 10x-new-flat
```

- Check configured secrets:

```powershell
& $wrangler secret list --name 10x-new-flat --format pretty
```

## Assumptions

- No custom domain for the first deploy.
- No Supabase migrations are needed because the repo has no `supabase/migrations`.
- Do not deploy without Supabase production secrets.
- First deploy has no previous Worker version to roll back to; use fix-forward or manually disable/delete the Worker if needed.
